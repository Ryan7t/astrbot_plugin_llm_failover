# 故障转移顺序配置化变更记录

## 问题概述
- 现有故障转移顺序默认以被调用的 Provider（primary）为首，其余 Provider 按框架返回顺序追加，无法通过插件配置重排首选提供商。
- 需要提供可视化配置入口，让运维可以显式定义首选、备选顺序，并在异常时严格按该顺序依次尝试。

## 备选方案与取舍
- 方案 A：保持 primary 在首位，仅允许配置备用顺序。缺点是无法彻底替换首选 Provider，无法满足“完全重排首选”的需求，放弃。
- 方案 B：新增插件配置 `priority_order`，只要填入 Provider ID 列表就完全按该顺序执行故障转移，未配置的 Provider 依次补齐。实现简单且满足要求，采用。
- 方案 C：依赖外部全局配置或在 Provider 层打补丁。会侵入核心逻辑且难以热更新，放弃。

## 实施步骤/关键改动
- 新增 `_conf_schema.json`，暴露 `priority_order` 列表配置，提示按 Provider ID 写入优先级，未填的 Provider 自动排在末尾。
- `main.py` 接收插件配置，新增 `_load_priority_ids` 兼容字符串/列表的解析逻辑，并通过 `_sort_providers` 统一执行顺序重排，配置存在时完全忽略 primary 的优先权。
- 故障转移安装日志同步输出自定义顺序，便于确认当前生效的调用链；无配置时保持原有 primary 优先行为。

## 优缺点与结论
- 优点：顺序可视化配置、可随时调整，满足“完全自定义首选 Provider”诉求；不破坏原有默认体验。
- 潜在缺点：需要准确填写 Provider ID，否则会被忽略并落到自动追加的末尾，需注意配置校验。
- 结论：通过插件配置完成故障转移顺序的彻底可配置化，首选 Provider 不再依赖框架 primary，而由 `priority_order` 决定。
